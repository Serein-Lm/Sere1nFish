# 在线客服话术 Skill

本 skill 专门处理在线客服渠道的社工话术。在线客服是一个特殊渠道——攻击者以"客户/用户"身份主动发起对话，客服人员有义务响应和帮助，这天然创造了一个高信任起点。

## 渠道特性分析

在线客服与其他渠道的核心区别：

| 特性 | 在线客服 | 电话 | 邮件 | 微信 |
|------|---------|------|------|------|
| 发起方 | 攻击者主动发起 | 攻击者主动拨打 | 攻击者主动发送 | 需要先添加好友 |
| 信任起点 | 高（客服有义务服务） | 中（需要自证身份） | 低（容易被忽略） | 低（需要通过验证） |
| 对话节奏 | 即时，可控 | 即时，但有时间压力 | 异步，不可控 | 即时，但需要好友关系 |
| 文件传输 | 部分支持 | 不支持 | 支持 | 支持 |
| 渠道切换 | 自然（"客服系统不方便，加个微信？"） | 自然（"我发您邮件"） | 不自然 | 不需要 |

## scenario 阶段

在 `LogicChainStep` 中，当 finding 为在线客服时：

### 推荐逻辑链条模板

**模板 A：客户咨询 → 技术问题 → 文件投递**
```
1. [live_chat] 以客户身份进入在线客服，咨询产品/服务问题
2. [live_chat] 描述一个"技术问题"，需要客服协助排查
3. [live_chat] 提供"问题截图/日志文件"（载荷），请客服帮忙查看
   fallback: 如果客服系统不支持文件传输 → 切换到微信/邮件
```

**模板 B：客户投诉 → 情绪施压 → 渠道切换 → 文件投递**
```
1. [live_chat] 以客户身份进入在线客服，反映产品问题/投诉
2. [live_chat] 表达不满，要求升级处理（转人工/转主管）
3. [live_chat] 要求加客服微信或获取邮箱，以便发送"问题证据"
4. [wechat/email] 发送"问题截图和日志"（载荷）
   fallback: 如果客服拒绝提供个人联系方式 → 要求工单系统上传附件
```

**模板 C：商务咨询 → 需求对接 → 方案交换 → 文件投递**
```
1. [live_chat] 以潜在客户身份咨询产品/服务
2. [live_chat] 表达采购意向，询问商务对接流程
3. [live_chat] 要求发送"我们的需求文档"给对方评估
4. [live_chat → wechat] 切换到微信发送"需求文档"（载荷）
   fallback: 如果无法切换渠道 → 在客服系统中发送文件链接
```

### 伪造身份建议

| 身份 | 适用场景 | 可信度 |
|------|---------|--------|
| 企业客户/采购负责人 | 目标是 B2B 公司 | 极高（客服必须认真对待） |
| 普通用户/消费者 | 目标是 C 端产品 | 高（客服有服务义务） |
| 技术人员/开发者 | 目标是技术产品/API 服务 | 高（技术问题需要深入沟通） |
| 合作伙伴/渠道商 | 目标有合作伙伴体系 | 中高（可能需要验证） |

## script 阶段

输出到 `ScriptOutput.scripts[].channel_scripts[]`，其中 `channel = "wechat"`（在线客服对话使用 wechat 的 DialogueTurn 格式）。

### DialogueTurn 填充规则

| 字段 | 规则 |
|------|------|
| `role` | `"attacker"` 或 `"target"`（target 在此场景中是客服人员） |
| `content` | 对话内容（口语化，符合在线聊天风格） |
| `tactic` | 使用的心理策略名称 |

### 在线客服对话的特殊要求

1. **对话不少于 8 个来回**（16 条 DialogueTurn）
2. **必须体现完整的信任建立过程**，不能一上来就发文件
3. **对话风格必须像真实的客服聊天**：
   - 使用口语化表达，不要书面语
   - 适当使用表情符号（但不要过多）
   - 消息长度适中（每条 1-3 句话，不要长篇大论）
   - 体现即时通讯的节奏感（有时一句话拆成两条发）

### 对话节奏模板（必须覆盖以下阶段）

```
阶段 1 - 开场（2-3 轮）
  攻击者：提出一个合理的咨询/问题
  客服：标准回应
  攻击者：进一步描述问题，展示"真实客户"的特征
  → tactic: 社会认同（我是你们的客户）

阶段 2 - 深入沟通（2-3 轮）
  攻击者：提供具体细节，让客服投入时间和精力
  客服：尝试帮助解决
  攻击者：表示问题比较复杂，需要更深入的沟通
  → tactic: 互惠原则（客服已经投入了时间，不好意思中断）

阶段 3 - 信任建立（1-2 轮）
  攻击者：对客服的帮助表示感谢，建立个人好感
  客服：继续提供帮助
  → tactic: 好感原则（让客服觉得这是个好客户）

阶段 4 - 需求植入（1-2 轮）
  攻击者：自然地引出"需要发送/接收文件"的需求
  客服：回应
  → tactic: 承诺一致性（客服已经在帮忙了，继续帮忙是一致的）

阶段 5 - 载荷投递（1-2 轮）
  攻击者：发送文件或引导客服到其他渠道接收文件
  客服：接收
  → tactic: 互惠原则 + 合理化（文件内容与之前的对话一致）
```

### 渠道切换话术（当客服系统不支持文件传输时）

| 切换目标 | 话术示例 |
|---------|---------|
| 微信 | "这个客服系统发文件不太方便，能加您微信吗？我把日志文件发您看看" |
| 邮件 | "我这边有个比较大的日志文件，能给个邮箱吗？我发过去您帮忙看看" |
| 工单 | "我在工单里上传了附件，麻烦您看一下，工单号是 XXXX" |

**关键：如果话术中出现了渠道切换，必须同时在 scripts 中提供切换后渠道的完整话术。**

### key_points 必须包含

- 进入客服系统的方式（点击哪个按钮/链接）
- 如果有机器人客服，如何快速转人工的话术
- 客服系统是否支持文件传输
- 渠道切换的时机和话术

## objection 阶段

在线客服场景必须覆盖的质疑类型：

1. "请问您的订单号/账号是？" → tactic: 模糊回应 + 转移话题（"我刚注册的，还没下单，想先了解一下"）
2. "这个问题需要您联系技术支持" → tactic: 坚持 + 互惠（"能帮我转接一下吗？我找了半天没找到入口"）
3. "我们不方便添加客户微信" → tactic: 合理化 + 退让（"理解的，那能给个技术支持邮箱吗？我把日志发过去"）
4. "请您通过工单系统提交" → tactic: 顺势而为（"好的，工单里可以上传附件吗？我把问题截图传上去"）
5. "这个文件我没办法打开" → tactic: 耐心 + 提供替代方案（"可能是压缩格式的问题，我换个格式发您"）
6. "您是我们的客户吗？我查不到您的信息" → tactic: 合理化（"我是用另一个账号注册的/我是帮同事问的"）

每条 `ObjectionItem` 的 `context_note` 必须说明与当前场景的具体关联。

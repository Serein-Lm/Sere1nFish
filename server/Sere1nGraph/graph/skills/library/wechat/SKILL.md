# 微信话术 Skill

本 skill 跨三个阶段复用。根据当前阶段填充对应的 Schema 字段。

**注意：本 skill 同时适用于微信对话和在线客服对话，因为两者的对话格式相同（即时通讯式的一问一答）。区别在于：**
- 微信场景：攻击者需要先添加好友，信任起点较低
- 在线客服场景：攻击者直接发起对话，信任起点较高（客服有服务义务）

## scenario 阶段

在 `LogicChainStep` 中，当 `channel = "wechat"` 或 `channel = "live_chat"` 时：
- `action` 描述微信/客服环节的具体动作（添加好友/发消息/发文件/咨询问题）
- `fallback` 考虑目标不通过好友申请/客服不配合的备选方案

## script 阶段

输出到 `ScriptOutput.scripts[].channel_scripts[]`，其中 `channel = "wechat"`。

### DialogueTurn 填充规则

| 字段 | 规则 |
|------|------|
| `role` | `"attacker"` 或 `"target"` |
| `content` | 对话内容（口语化，符合即时通讯风格） |
| `tactic` | 使用的心理策略名称（互惠原则/权威效应/紧迫感/社会认同/承诺一致性/损失厌恶等） |

### 对话基本要求

- 对话不少于 **8 个来回**（16 条 DialogueTurn）
- 包含完整的信任建立过程：寒暄 → 业务切入 → 信任建立 → 需求植入 → 文件引导 → 确认打开
- `key_points` 填入关键要点（好友验证消息建议、备注名建议等）

### 对话风格要求

- **必须口语化**：像真人聊天一样，不要书面语
- **消息长度适中**：每条 1-3 句话，不要长篇大论
- **节奏自然**：有时一句话拆成两条发，有时等对方回复再继续
- **适当使用语气词**：嗯、好的、哈哈、了解、收到
- **禁止出现**：过于正式的措辞、不自然的长句、明显的模板痕迹

### 渠道切换约束（极其重要）

**如果对话中攻击者提到了任何其他渠道，必须同时提供该渠道的完整内容：**

| 对话中提到 | 必须同时提供 |
|-----------|------------|
| "我发邮件给您" | scripts 中必须有 channel=email 的完整 email_template |
| "我打电话给您" | scripts 中必须有 channel=phone 的完整电话对话 |
| "加我微信" | scripts 中必须有 channel=wechat 的完整微信对话（如果当前不是微信渠道） |
| "我在工单里提交" | key_points 中必须说明工单提交的具体内容 |

**绝不允许"提到但不提供"。如果你不打算提供某个渠道的完整内容，就不要在对话中提到它。**

### 心理策略

| 策略 | 用法 | 在对话中的体现 |
|------|------|--------------|
| 互惠原则 | 先提供价值再请求 | 先帮对方解决问题，再请求对方做某事 |
| 权威效应 | 利用职位/公司背书 | 提及公司名、职位、项目名增加可信度 |
| 紧迫感 | 制造时间压力 | "截止日期快到了""这个问题比较紧急" |
| 社会认同 | 提及其他人也在做 | "您同事也是这样操作的""其他客户都是这么处理的" |
| 承诺一致性 | 利用对方之前的承诺 | 从小请求逐步升级到大请求 |
| 损失厌恶 | 强调可能的损失 | "如果不处理可能会影响XX""错过这次要等半年" |

详细案例见 `references/wechat-dialogue-cases.md`。

## objection 阶段

输出到 `ObjectionOutput.scenario_objections[].objections[]`。

微信场景必须覆盖的质疑类型：
1. "你怎么加到我微信的？" → tactic: 社会认同+权威效应
2. "我不加陌生人" → tactic: 退让策略+互惠原则
3. "这个文件我不敢打开" → tactic: 消除顾虑+提供验证途径
4. "为什么要用压缩包？" → tactic: 合理化+权威效应
5. "我先问下领导/同事" → tactic: 紧迫感+权威效应
6. "你能证明你的身份吗？" → tactic: 提供可验证信息（工号、前台电话、官网信息）

每条 `ObjectionItem` 的 `context_note` 必须说明与当前场景的具体关联。

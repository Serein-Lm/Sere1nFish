# 邮箱话术 Skill

本 skill 跨三个阶段复用。根据当前阶段填充对应的 Schema 字段。

## scenario 阶段

在 `LogicChainStep` 中，当 `channel = "email"` 时：
- `action` 描述邮件环节的具体动作（发送方案/报价/通知）
- `fallback` 考虑邮件被忽略/进垃圾箱的备选（电话跟进）

## script 阶段

输出到 `ScriptOutput.scripts[].channel_scripts[]`，其中 `channel = "email"`。

### ChannelScript 填充规则

| 字段 | 规则 |
|------|------|
| `channel` | 固定 `"email"` |
| `email_template` | **必填** — 完整邮件模板（含发件人/主题/正文/签名） |
| `dialogue` | 可选 — 如果有邮件往来对话 |
| `key_points` | 发件人邮箱格式建议、主题命名规范等 |

### email_template 格式要求

```
发件人: {伪造姓名} <{伪造邮箱}>
主题: {与业务上下文一致的主题}

{称呼}，您好：

{正文 — 简洁专业，符合行业语言习惯，不少于 150 字}

{附件说明 — 自然地提及压缩包和密码}

此致
敬礼

{伪造姓名}
{伪造公司} | {伪造职位}
电话：{伪造电话}
邮箱：{伪造邮箱}
```

### 邮件正文要求（强化）

邮件正文必须包含以下要素，且不少于 150 字：
1. **背景说明** — 为什么发这封邮件（与之前的沟通/业务需求关联）
2. **核心内容** — 邮件的主要信息（方案说明/问题描述/合作意向）
3. **附件说明** — 自然地提及附件内容和加密原因
4. **行动引导** — 希望对方做什么（查看附件/回复确认/安排会议）
5. **签名完整** — 姓名、职位、公司、电话、邮箱缺一不可

### 关键规则
- 称呼准确（使用目标真实姓名/职位）
- 附件说明自然（不要刻意强调"请打开"）
- 签名完整（姓名、职位、公司、电话、邮箱）
- 提供合理理由解释为什么需要加密
- **邮件内容必须与场景上下文完全一致**（如果是从电话/客服对话中切换过来的，邮件开头必须提及之前的沟通）

### 渠道切换场景的邮件模板

当邮件是从其他渠道切换过来时（如电话后发邮件、客服对话后发邮件），邮件开头必须引用之前的沟通：

```
{称呼}，您好：

刚才{电话/在线}跟您沟通过{具体事项}，按照约定把{文件内容}发您。

{后续内容...}
```

详细案例见 `references/email-template-cases.md`。

## objection 阶段

输出到 `ObjectionOutput.scenario_objections[].objections[]`。

邮箱场景必须覆盖的质疑类型：
1. 目标回复"这封邮件是真的吗？" → tactic: 提供验证途径
2. 目标未回复（已读不回） → tactic: 紧迫感跟进
3. 目标回复"附件打不开" → tactic: 提供解决方案 + 引导到微信渠道
4. 目标转发给 IT 部门审查 → tactic: 渠道切换
5. 目标回复"为什么加密？" → tactic: 合理化（商业保密/公司规定/包含敏感信息）

每条 `ObjectionItem` 的 `context_note` 必须说明与当前场景的具体关联。

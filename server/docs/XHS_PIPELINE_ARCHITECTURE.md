# 小红书社工信息采集系统 - 架构与概念解析

## 系统概述

本系统用于自动化采集和分析小红书平台上与特定关键词（如公司名称）相关的社会工程学攻击面信息。

**核心目标**：通过搜索关键词，找到与目标公司/组织相关的人员，分析其信息暴露情况，生成可用于安全评估的人物画像。

---

## 核心概念解析

### 1. Cookie 激活机制

#### 什么是 Cookie 激活？

小红书爬取需要有效的登录态（Cookie）。系统支持存储多个账号的 Cookie，但**同一时间只能使用一个账号进行爬取**。

```
账号1 (Cookie A) ──┐
账号2 (Cookie B) ──┼── 只有一个是激活状态 ──> 用于爬取请求
账号3 (Cookie C) ──┘
```

#### 为什么需要多账号？

1. **风控规避**: 单账号频繁请求容易触发平台风控
2. **账号轮换**: 一个账号被限制后可切换到其他账号
3. **并行采集**: 不同项目可使用不同账号（未来特性）

#### API 操作说明

| 操作 | 说明 |
|------|------|
| 添加账号 | 存储新的 Cookie，默认非激活状态 |
| 激活账号 | 将账号设为当前使用，其他账号自动变为非激活 |
| 验证账号 | 检测 Cookie 是否仍然有效 |
| 删除账号 | 移除账号及其 Cookie |

### 2. 搜索关键词与关联度

#### 关键词的核心作用

搜索关键词（如"字节跳动"）是整个采集流程的**核心锚点**：

```
搜索关键词: "字节跳动"
    │
    ├── 笔记A: 直接提及字节跳动 → 关联度 95
    ├── 笔记B: 讨论抖音产品 → 关联度 80 (子公司/产品)
    ├── 笔记C: 讨论腾讯 → 关联度 30 (竞争对手)
    └── 笔记D: 讨论美食 → 关联度 5 (无关)
```

#### 关联度评估 (keyword_relevance: 0-100)

**评分维度**:

| 分数段 | 关联程度 | 典型场景 |
|--------|----------|----------|
| 90-100 | 直接相关 | 笔记正文明确提及搜索关键词 |
| 70-89 | 高度相关 | 提及子公司/产品/部门（如抖音属于字节） |
| 50-69 | 中度相关 | 同行业但不同公司（如讨论互联网大厂） |
| 20-49 | 弱相关 | 仅行业层面沾边 |
| 0-19 | 无关 | 与搜索词无任何关联 |

#### 为什么关联度重要？

**过滤噪声**: 小红书搜索结果混杂，很多笔记只是蹭热度但与目标无关

**精准定位**: 高关联度的笔记更可能来自目标公司的真实员工

**评分权重**: 关联度直接影响最终的 attention_score

### 3. 公司归属识别

#### 核心问题

> 这个用户是否属于我们搜索的目标公司？

#### 判断依据

1. **直接声明**: "我在字节工作"、"作为抖音的员工"
2. **昵称暗示**: "小明@字节跳动"、"抖音产品经理"
3. **内容推断**: 讨论内部系统、使用内部术语
4. **交叉验证**: 多篇笔记信息互相印证

#### 数据结构

```json
{
    "company_identified": {
        "name": "字节跳动",
        "confidence": "high",
        "evidence": [
            "昵称包含'抖音'",
            "笔记多次提及字节内部系统"
        ],
        "related_to_keyword": true,
        "relationship_type": "等于搜索关键词"
    }
}
```

#### 关系类型说明

| relationship_type | 含义 | 示例 |
|-------------------|------|------|
| 等于搜索关键词 | 用户属于搜索的公司 | 搜"字节"，用户是字节员工 |
| 子公司 | 用户属于搜索公司的子公司 | 搜"字节"，用户是抖音员工 |
| 投资公司 | 用户属于被投资的公司 | 搜"腾讯"，用户是拼多多员工 |
| 竞争对手 | 用户属于竞品公司 | 搜"字节"，用户是快手员工 |
| 无关 | 与搜索词无明确关联 | - |

### 4. 三层打标架构

系统采用三层递进的 AI 分析架构：

```
                    ┌─────────────────────────────────────┐
                    │       第一层：搜索结果打标          │
                    │   输入: 标题 + 简介 + 用户信息      │
                    │   输出: 关联度 + 初步攻击面识别     │
                    └──────────────┬──────────────────────┘
                                   │
                          筛选 (is_suspicious = true)
                                   │
                    ┌──────────────▼──────────────────────┐
                    │       第二层：笔记详情打标          │
                    │   输入: 笔记全文 + 评论             │
                    │   输出: 深度关联度 + 公司识别       │
                    │         + 详细发现列表             │
                    └──────────────┬──────────────────────┘
                                   │
                          按用户聚合
                                   │
                    ┌──────────────▼──────────────────────┐
                    │       第三层：人物画像生成          │
                    │   输入: 用户所有笔记的分析结果      │
                    │   输出: 综合画像 + 攻击向量         │
                    │         + 公司归属判定             │
                    └─────────────────────────────────────┘
```

#### 第一层：搜索结果打标

**目的**: 快速过滤，识别值得深入分析的笔记

**输入**: 
- 笔记标题、简介
- 用户昵称、头像
- 搜索关键词

**输出**:
- `keyword_relevance`: 关联度评分
- `is_suspicious`: 是否可疑（需深入分析）
- `attention_score`: 关注度评分
- `company_mentioned`: 提及的公司名

#### 第二层：笔记详情打标

**目的**: 对可疑笔记进行深度分析

**输入**:
- 笔记完整内容
- 评论信息
- 搜索关键词

**输出**:
- `keyword_relevance`: 深度关联度评估
- `keyword_analysis`: 关联性详细分析
- `company_identified`: 公司归属识别
- `findings`: 敏感信息发现列表

#### 第三层：人物画像生成

**目的**: 聚合用户信息，生成完整画像

**输入**:
- 用户所有可疑笔记的分析结果
- 搜索关键词

**输出**:
- `keyword_relevance`: 综合关联度
- `company_identified`: 最终公司归属判定
- `profile_summary`: 人物画像描述
- `potential_attack_vectors`: 可能的攻击向量
- `recommended_actions`: 建议后续行动

### 7. 评分体系详解

#### is_suspicious（是否可疑）

**判定条件**: `keyword_relevance >= 50 且 attention_score >= 40`

只有同时满足两个条件的笔记才会进入深度分析阶段。

#### attention_score（关注度评分）

**计算逻辑**:

```
基础分 = keyword_relevance × 0.4~0.5

加分项:
+ 员工身份泄露 (employee_leak): +15~25
+ 凭证信息泄露 (credential_leak): +15~25
+ 内部信息泄露 (insider_info): +12~20
+ 联系方式暴露 (contact_info): +10~15
+ 组织架构信息 (org_structure): +10~15
+ 位置信息 (location_info): +4~5
...

上限: 100分
```

**意义**: 衡量这条信息对社工攻击的价值

#### keyword_relevance（关键词关联度）

专门衡量内容与搜索关键词的关联程度。

**为什么单独设置？**
- 搜索结果可能包含大量无关内容
- 即使信息泄露严重，如果与目标无关也没有价值
- 关联度是筛选的第一道门槛

## 常见问题

### Q: 为什么关联度和关注度是分开的？

**A**: 这是两个独立的评估维度：
- **关联度**: 内容是否与搜索目标相关
- **关注度**: 信息是否有利用价值

一条笔记可能高关联但低风险（如只是普通地提到公司），也可能低关联但有参考价值（如讨论行业通用的安全问题）。

### Q: attention_threshold 阈值如何设置？

**A**: 建议：
- **60（默认）**: 平衡精度和召回
- **40-50**: 较宽松，会返回更多可能相关的结果
- **70-80**: 较严格，只返回高价值目标

### Q: 公司归属识别的可信度如何理解？

**A**: 
- **high**: 有明确证据（如直接声明）
- **medium**: 有较强线索但非直接证据
- **low**: 推测性判断，需人工核实

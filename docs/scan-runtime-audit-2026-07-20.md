# 扫描运行审计与恢复方案（2026-07-20）

## 结论

本次长任务停滞不是单一资源不足，而是四个问题叠加：开发热重载反复中断进程、URL 结果缺少逐条检查点、确定性超时仍触发昂贵重试、公众号深采没有任务级总量上限。旧 Chrome 池确实偏小，但只扩池会同步放大无效 Agent 调用和 Token 成本。

本轮已把恢复粒度收敛到“单 URL、单来源模块、核心阶段、手机阶段”。第一阶段将 Chrome 上限从 16 提高到 64；启动审计发现 Docker/npm/FOFA 和手机网络存在瞬时启动风暴后，第二阶段将按需上限提高到 80，并补齐独立的创建与健康检查限流。后端在完成测试前保持停止；完成所有验证后只启动一次。

## 根因证据

### 1. 热重载重复执行

- 停机前日志在约三小时内出现 7 次应用初始化。
- 北京歌华和大连商品交易所的根来源阶段各启动了 3 次。
- 重载会销毁 Provider 进程状态并断开在用 CDP；旧实现只能在父任务结束时形成完整结果，导致已完成页面重新扫描。

### 2. URL 失败放大成本

- 旧流程在 URL 批次结束后统一写摘要，单页成功没有可靠恢复边界。
- Agent 内部失败与 stream stage 重试叠加；超时已经耗尽 100-120 秒预算，再重试通常没有收益。
- WAF、HTTP 错误页和 Nginx 默认页也进入 Agent，产生无效浏览和结构化修复。

### 3. Chrome 容量与连接稳定性

- 旧配置为 `max=16 / bulk=14 / reserve=2`，日志多次达到 13-15 个活跃容器。
- 业务日志出现 53 次 `Could not connect to Chrome`，并有 6 个实际 CDP 健康恢复事件。
- 未发现 OOM；当前服务器 32 核、可用内存约 51 GiB、load 约 2.3，存在明显余量。
- CDP 异常与多次应用重载高度重合，不能全部归因于服务器资源不足。

### 4. 公众号深采失控

- 东方有线产生 144 条手机记录、813 个截图引用，但只有 1 个唯一真实文章文档。
- 7 月 19 日观测记录为复制链接成功 29 次、失败 107 次；主要错误是剪贴板未更新或当前页面已不在微信前台。
- 旧坐标点击在页面布局变化后仍持续尝试；深采上限只约束单屏/单关键词，无法限制整个公司任务。

### 5. 恢复关联错误

- 招投标归档使用 `${task_id}_bidding`，视觉分析使用 `${task_id}_bidding_visual`。
- 旧恢复代码错误查询父任务 ID 与 `${task_id}_visual`，会把已有招投标恢复成 0 条。
- 根来源完成后等待手机时，尚未执行的全资子公司 follow-up 会被 `core_completed` 误跳过。当前 23 个检查点中有 13 个父目标、合计 51 个子公司需要继续处理。

## 本轮修复

### URL 与网站

- 每条 URL 成功或最终失败后立即按稳定 ID 幂等 upsert。
- 同一任务重启时跳过已有终态 URL，并从数据库汇总成功、失败、跳过和剩余数量。
- 最终汇总强制验证所有存活 URL 均有持久化终态；缺一条就保持可恢复错误，不再假完成。
- 分析总超时不做第二次昂贵重试。
- 复用 FOFA/Hunter 探活状态、标题和指纹，不在深扫入口重复探活。
- HTTP 错误、WAF 和通用默认页直接形成无 Finding 终态。
- 浏览器工具预算最终设为 6 次，输出仍使用结构化 schema 与纠正提示；资产分诊输出截断时自动拆半并注入纠正提示重试。

### 公司与任务恢复

- 单来源完成立即保存模块检查点。
- 根来源和手机阶段分别标记；无需手机任务时在一次 Mongo 更新中同时写入两个阶段，消除中断窗口。
- 公众号后台任务完成后自行写入手机阶段标记；根网站/API 一结束就释放公司核心配额并继续排队执行子公司 follow-up，最终保存综合结果前才等待并合并手机结果。
- `checkpoint.modules.wechat` 优先于阶段标记恢复；即使进程恰好在公众号检查点写入后、`mobile_completed` 写入前重载，也不会重复操作手机。
- 23 条根来源检查点从资产、URL、招投标、学者等持久化集合重建摘要，不重复调用 FOFA、Hunter、天眼查和网站 Agent。
- 修正招投标子任务 ID；未完成的 51 个子公司 follow-up 继续执行，已存在 follow-up 检查点则复用。
- 重排队只更新 `progress.stage/message`，不再覆盖各来源子进度。
- 学者检索自动复用已有英文/ASCII 公司别名。

### Chrome 与并发

- Chrome：`max=80`、批量额度 72、非批量保留 8、预热 20。
- Docker 容器创建最多 4 路，预热池每次只补一波，避免后台预热长期占住业务创建队列。
- Docker stats/CDP 健康检查最多 12 路，避免 80 个容器每 30 秒同时冲击 Docker API。
- URL Agent worker：默认 24、可配上限 48。
- URL/资产探活：128；话术：12；公司核心并发：6。
- 容量可运行时 resize，不替换现有租约。
- 手机队列等待不占公司核心配额；同一目标的子公司、网站和 API 后续可在公众号慢速采集期间继续推进。
- 主机保护：可用内存低于 8 GiB、每核 load 超过 0.9、1 分钟内近期 CDP 异常达到 12 次时停止新建容器；已有任务不被强杀。
- 兼容旧 Mongo 字段，避免新代码静默退回默认保护参数。

### 公众号与手机

- 公司公众号最多 6 个关键词、每关键词初始屏加 3 次滑动、详情最多滑 6 屏、总运行 30 分钟。
- 新增任务级深采总上限 2，跨关键词和跨屏去重。
- UIAutomator 优先定位“更多/复制链接”，固定坐标仅作兼容回退。
- 导航未明确完成不再静默成功，而是进入可观测重试/失败分支。
- 超时返回 partial，保留已入库记录、截图、真实链接和文章归档。

### 通知与前端

- 多公司只在批次终态发送一条摘要；无高价值且无失败时不发。
- 每家公司不再逐条通知普通完成；只展示高分结果和最多 5 个优先目标。
- 浏览器同一用途“已恢复”10 分钟最多一条，“恢复失败”5 分钟最多一条；详细事件仍保留在日志。
- 钉钉时间固定为 `Asia/Shanghai`。
- 网站列表与看板统一读取 URL 扫描结果，保持高分优先；增加 Target 筛选。
- 任务列表和详情展示网站、招投标、公众号等来源进度；paused/cancelled 不再误显示成等待并持续轮询。

## 第二阶段启动审计

第一次恢复运行在 2026-07-20 02:00（北京时间）调度了批次 `4eafabcb1b2e` 的 105 条 pending，5 条 completed 未重新认领，23 条核心检查点被复用。运行约 3 分钟后受控停止，所有未完成任务重新回到 pending，来源检查点和 URL 终态均保留。

日志确认了以下启动期问题及处理：

- EasyTier/ADB 晚于业务恢复约几十秒就绪，首个公众号任务把稳定设备 ID 当成 ADB endpoint。手机任务现在最多等待 120 秒，只有解析出在线 endpoint 后才认领定义并触发“已开始”。
- 多个 Agent 同时执行 `npx chrome-devtools-mcp@latest`，共享 npm 缓存出现 `TAR_ENTRY_ERROR`。后端镜像固定安装 `chrome-devtools-mcp@1.6.0`，运行时不再执行 npx 下载。
- 6 家公司各自的 FOFA Provider 只做实例内节流，触发 `[45012] 请求速度过快`。FOFA 请求起点现在在进程内跨 Provider 共享 2 秒节流门，重试同样经过该门。
- 业务容器与 16 个预热容器同时创建时 Docker API 阻塞，旧 Compose 10 秒停机窗口最终产生 Exit 137 和残留 Chrome。创建限流统一收敛到 Provider，取消中的不可中断 Docker 调用会等待返回后删除容器；停机并行回收，Compose 宽限期提高到 120 秒。
- nginx upstream 在启动时缓存旧 backend IP，backend recreate 后持续 502，必须手工 reload。nginx 1.27 现在通过 Docker DNS `127.0.0.11` 动态解析 backend/frontend，容器换 IP 后自动刷新。
- 公众号定义在多条恢复任务中被并发重复修复 8 次。定义的创建、选择和修复按 project/device 共用同一锁，只有第一条写库。
- 资产 LLM 分诊一次返回 30 条时在第 27 条附近被截断。默认批次降为 20，格式失败后拆半并带纠正提示重试；重试仍失败时继续保留资产，不丢 URL。
- 正常服务关闭导致 stream pipeline 收到 `CancelledError`，旧日志误记为 ERROR。现在按正常取消记录 info，同时仍执行 worker 回收；真实异常继续记 ERROR。
- 单元测试中的 ADB server 异常曾在源码挂载目录生成 32 MiB core dump。后端容器已禁用 core dump，异常仍保留日志和退出码，不再污染工作区或占用磁盘。
